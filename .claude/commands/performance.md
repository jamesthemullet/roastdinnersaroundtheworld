You are running a performance review session for this Astro + React project.

## Stack

- **Astro 6** with React 19 islands (`client:only="react"`)
- **TypeScript** — not in strict mode; uses Astro's base tsconfig (`astro/tsconfigs/base`)
- **Plain CSS** — per-component/per-page `.css` files; no Tailwind, no Emotion
- **Vitest** + **@testing-library/react** for unit tests; **Playwright** for E2E
- Source files live in: `src/components/`, `src/layouts/`, `src/lib/`, `src/pages/`
- Data fetched from a WordPress GraphQL API (`src/lib/api.ts` → `fetchGraphQL`)
- Currency conversion via exchangerate-api.com at build/request time

## What to do each invocation

### Step 1 — Audit for performance issues

Read the source files in `src/components/`, `src/layouts/`, `src/lib/`, and `src/pages/`. Look across these categories:

1. **Data fetching** — redundant or repeated `fetchGraphQL` / `fetch` calls on the same page or across pages that could be deduplicated; missing `async` parallelism (sequential awaits that could be `Promise.all`); uncached external API calls (e.g. exchange rates fetched on every request instead of at build time)
2. **Bundle size / JavaScript** — large dependencies imported in full when only a subset is needed; client-side React islands doing work that could be done at build time in Astro; unnecessary `client:only` or `client:load` directives on components that don't need interactivity
3. **Rendering** — Astro pages using server-side rendering (`output: 'server'`) for pages that are fully static and could be pre-rendered; missing `preload` / `prefetch` hints for critical resources; large unoptimised images without `width`/`height` or without the Astro `<Image>` component
4. **React component efficiency** — missing `useMemo` / `useCallback` on expensive derivations or callbacks passed as props; `useEffect` with no or wrong dependency arrays causing unnecessary re-renders; state initialised from props without stabilisation
5. **CSS / layout** — render-blocking CSS that could be inlined or deferred; large unused CSS rules

### Step 2 — Classify each finding

For each finding, assign a severity:

- **Minor** — small, localised gain; low effort; low risk (e.g. adding `Promise.all`, fixing a missing `useMemo`)
- **Significant** — meaningful impact on load time, bundle size, or runtime performance; warrants its own focused issue

Discard any finding that is purely theoretical with no evidence of actual cost in this codebase.

If you find **nothing worth improving**, stop here and report that clearly. Do not invent issues.

### Step 3 — Report your findings

Output exactly this structure:

```
## Performance review

**Date:** <today's date>

### Significant findings
<numbered list — or "None" if there are none>

### Minor findings
<numbered list — or "None" if there are none>

### Verdict
<one sentence: e.g. "Two significant issues and three minor issues found." or "No meaningful performance improvements identified.">
```

### Step 4 — Create GitHub issues

**If there are NO findings at all:** Do not create any issues. Report the verdict and stop.

**If there are ONLY minor findings:** Create a single GitHub issue grouping them all:

```bash
gh issue create \
  --title "perf: minor performance improvements" \
  --label "performance" \
  --body "## Minor performance improvements

The following small, low-risk performance improvements were identified during a routine review.

<bulleted list of findings, each with file path and a one-sentence description of the fix>

---
*Generated by the performance skill.*"
```

**For each SIGNIFICANT finding:** Create one issue per finding:

```bash
gh issue create \
  --title "perf: <short title>" \
  --label "performance" \
  --body "## Summary

<one paragraph describing the issue and why it matters>

## Location

`<file path>` — <relevant function or section>

<concrete description of what to change>

## Expected impact

<what measurable improvement is expected: e.g. reduced waterfall requests, smaller JS bundle, fewer re-renders>
```

Report all issue URLs once created.

## Known project patterns

- **Data fetching:** `fetchGraphQL(query, variables?)` in `src/lib/api.ts` — raw `fetch` to the same host is a smell
- **GraphQL queries:** Live in `src/lib/graphql/queries/` as named exports
- **React island:** `SortPosts` in `src/components/sortPosts.tsx` is the only interactive React component; it uses `client:only="react"` in `league-of-roasts.astro`
- **Currency normalisation:** Exchange rates are fetched in `league-of-roasts.astro` at request/build time — this is intentional; flag if it regresses to client-side fetching
- **`console.error`** in catch blocks is intentional; ignore it
- **No router:** Astro pages are file-based; no client-side navigation overhead to worry about
