You are running a security review session for this Astro + React project.

## Stack

- **Astro 6** with React 19 islands (`client:only="react"`)
- **TypeScript** — not in strict mode; uses Astro's base tsconfig (`astro/tsconfigs/base`)
- **Plain CSS** — per-component/per-page `.css` files; no Tailwind, no Emotion
- **Vitest** + **@testing-library/react** for unit tests; **Playwright** for E2E
- **ESLint** (flat config) + **Prettier** + **Knip**
- Source files live in: `src/components/`, `src/layouts/`, `src/lib/`, `src/pages/`
- Data fetched from a WordPress GraphQL API (`src/lib/api.ts` → `fetchGraphQL`)
- Currency conversion via exchangerate-api.com at build/request time

## What to do each invocation

### Step 1 — Audit for security issues

Read the source files in `src/components/`, `src/layouts/`, `src/lib/`, and `src/pages/`. Also read `package.json`, any `.env*` files present, `astro.config.*`, and CI/CD config files (`.github/workflows/`). Look across these categories:

1. **Secrets and credentials** — API keys, tokens, or credentials hardcoded in source files or committed `.env` files; environment variables exposed to the client that should stay server-side only (check for `PUBLIC_` prefix on sensitive vars in Astro); secrets visible in git history
2. **Injection vulnerabilities** — unsanitised user input passed to `innerHTML`, `dangerouslySetInnerHTML`, `eval`, `Function()`, or injected into GraphQL queries; template literals constructing GraphQL queries with user-supplied values instead of using variables
3. **Dependency vulnerabilities** — run `npm audit --json` and parse the output; flag any `high` or `critical` severity advisories with the affected package, CVE/advisory ID, and recommended fix version
4. **HTTP security headers** — check `astro.config.*` and any middleware for missing or misconfigured security headers: `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `Strict-Transport-Security`
5. **Data exposure** — GraphQL responses that return more fields than the page consumes (over-fetching sensitive data); API responses logged to `console.log` in production code; error messages that leak internal paths or stack traces to the client
6. **External resource integrity** — third-party scripts or stylesheets loaded without `integrity` (SRI) attributes; external fetch calls without timeout or error handling that could expose internal failures
7. **CORS and request validation** — any API routes or server endpoints that accept cross-origin requests without proper validation; missing input validation on query parameters used in data fetching

### Step 2 — Classify each finding

For each finding, assign a severity:

- **Critical** — directly exploitable or exposes credentials/user data (e.g. hardcoded API key, XSS via `dangerouslySetInnerHTML` with user input, high/critical npm advisory)
- **Significant** — weakens security posture or enables indirect attack paths (e.g. missing CSP header, GraphQL query built from string interpolation, sensitive data logged)
- **Minor** — best-practice improvement with low direct risk (e.g. missing `X-Content-Type-Options`, SRI missing on a low-privilege script, over-fetching non-sensitive fields)

Discard any finding that is purely theoretical with no evidence of an actual issue in this codebase. Do not invent issues.

If you find **nothing worth improving**, stop here and report that clearly.

## Known project patterns

- **API base URL:** Stored in `src/lib/api.ts` as `API_URL` — must not be a `PUBLIC_` Astro env var since it's server-side only; this is correct if already server-only
- **fetchGraphQL:** All WordPress GraphQL calls go through `fetchGraphQL` in `src/lib/api.ts` — raw `fetch` to the same host bypassing this helper is a smell (potential for missed error handling or header stripping)
- **GraphQL queries:** Live in `src/lib/graphql/queries/` as static named exports — inline template literals constructing queries from user input would be a significant finding
- **React island:** `SortPosts` in `src/components/sortPosts.tsx` is the only interactive React component; client-side state comes from props passed at build time, not from user-supplied URLs
- **`console.error`** in catch blocks is intentional and acceptable; `console.log` with response data in non-catch paths is the smell to flag
- **No auth or user accounts:** This is a public read-only site; there is no login, session handling, or user-submitted content — do not flag absence of auth as a finding
- **Exchange rate API:** Called via `fetch` at build/request time in `league-of-roasts.astro`; flag if the API key is hardcoded or if a failure leaks an internal error message to rendered HTML
